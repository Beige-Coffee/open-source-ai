(Reading: The Open Source AI Definition v1.0) is the first formal
definition of "open source AI" published by a recognized body. It
landed in October 2024 after a multi-year drafting process by the
Open Source Initiative. It is contested in specific ways. The v2.0
revision cycle, scheduled to overlap Q4 2026 (the Sovereign AI Summit
window), is where the contested clause gets revisited.

This essay names the fight clearly so you can have a position when
the cycle starts.

## The contested clause

OSAID v1.0 requires four things for a model to be called open source
AI: usable for any purpose, studyable, modifiable, shareable. These
mostly translate cleanly from open-source software.

The contested fifth thing is **what counts as making the system
"studyable" and "modifiable."** OSAID v1.0 settled on this language:

> "Sufficient detail about the data and the training process such
> that a skilled person can recreate a substantially equivalent
> system."

Two camps read this differently.

## The Maffulli camp (loose data, pragmatic)

Stefano Maffulli is OSI's executive director. The Maffulli reading:
"sufficient detail about the data" allows publishing a thorough data
description (sources, methodology, processing pipeline, filtering
criteria) without publishing the raw training data itself.

The pragmatic case: full training-data disclosure for a frontier
model is essentially impossible under current copyright law. A
typical frontier pretraining corpus contains licensed content
(news, books, academic publications), web-scraped content of
ambiguous license status, and content the lab cannot republish even
if it wanted to. Requiring full data disclosure sets a definition
that no frontier-class model can meet, which makes the definition
irrelevant.

Maffulli's position: a definition that excludes 99% of the field is
not a useful definition.

## The Kuhn camp (strict data, principled)

Bradley Kuhn is at the Software Freedom Conservancy. The Kuhn
reading: "sufficient detail" without the actual data is not
sufficient. You cannot study what you cannot see. You cannot modify
what you cannot retrain. A definition that allows closed-data models
to claim open-source status erodes the meaning of "open source."

The principled case: open-source software has worked precisely
because the source is available, not because the source is described.
Code review depends on the actual code. Training-data review depends
on the actual data. If frontier models cannot meet that bar, the
right answer is to call them something other than open source AI,
not to weaken the definition until they fit.

Kuhn's position: a definition that softens to fit the frontier is
not a definition; it is marketing.

## What v1.0 actually does

OSAID v1.0 takes the Maffulli line. The published definition allows
"data information sufficient" rather than "data published." This is
the operative compromise.

Practical consequence: under OSAID v1.0, (Project: olmo) qualifies
because AI2 publishes data, code, weights, and training logs. (Project:
llama-3) does not qualify because the license is restrictive (see
(Essay: llama-license-operative-constraint)) and the data is not
disclosed. (Project: deepseek-v3) is ambiguous because the license is
open but the data disclosure is not what AI2 does.

So OSAID v1.0 is strict enough to exclude Llama, lenient enough to
admit DeepSeek under some readings, and consistent with OLMo.

## Where v2.0 will land

The v2.0 cycle is officially open. The substantive options:

**Option A: tighten toward Kuhn.** Require actual data publication or
explicit data lineage with permissive licensing. OLMo qualifies;
DeepSeek probably does not; nothing from the major US or Chinese labs
qualifies. Definition becomes meaningful but applies to almost
nothing.

**Option B: keep v1.0 line.** Continue with "data information
sufficient." Makes OLMo the gold standard but allows looser releases
to claim the label.

**Option C: loosen further.** Allow source-available weights with
restrictions to qualify if they include sufficient documentation.
Probably brings Llama and Gemma in; collapses the distinction
between open source AI and source-available AI.

The risk if v2.0 loosens: "open source AI" stops meaning anything
specific, and the term goes the way of "natural" in food labeling.
The risk if v2.0 tightens: the term applies to a small set of
research-grade releases and the broader market ignores it.

## Why this matters operationally

The Sovereign AI Summit (Fall 2026) overlaps the OSAID v2.0
finalization window. That timing is not coincidental for the cypherpunk
framing of the summit: the sovereignty argument requires a definition
of "open" that is meaningfully open. If v2.0 loosens, the summit's
panel on open-source-AI definitions becomes a panel about whether the
official definition still means anything. If v2.0 tightens, the panel
becomes about whether the official definition can be operationalized
at scale.

There is also a downstream effect on grant funding. (Funder: nlnet),
(Funder: mozilla), and several others fund "open-source AI"
explicitly. Their grant criteria have to track the definition. If the
definition loosens, the bar lowers for what counts as a fundable
open-source-AI project. If it tightens, the bar raises and the eligible
project pool shrinks.

## The position this site takes

Pragmatic in language, strict in tagging. We use "open weights" for
the loose category (anything you can download), "OSAID-compliant" for
the strict category (passes v1.0), and "fully open" for the strictest
(OLMo-style data + code + weights). This avoids the hard fight and
lets the reader pick their own threshold.
