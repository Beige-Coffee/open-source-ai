This essay exists because the silicon-as-sovereignty argument lives or
dies at the consumer hardware question, and the consumer hardware
question has a specific technical shape that surprises most people.

## The mental model

To predict one token, an LLM does a forward pass through every weight
in the model. The compute is fast. The compute is always fast. The
bottleneck is shoveling weights from memory into the compute units.

So the absolute upper bound on tokens per second at batch size 1 is:

```
max_tokens_per_second = memory_bandwidth / model_size_in_bytes
```

That is a hard ceiling. Compute cannot help you go faster than this,
because the weights physically have to arrive at the multiplier
before the multiplier can do anything with them.

## Concrete numbers

Memory bandwidth in 2026:

| Hardware | Bandwidth |
|---|---|
| H100 (HBM3) | ~3,350 GB/s |
| RTX 4090 (GDDR6X) | ~1,008 GB/s |
| Apple M4 Max | ~546 GB/s |
| Apple M2/M3 Ultra | ~800 GB/s |
| Apple M4 Pro | ~273 GB/s |
| Strix Halo / Ryzen AI Max+ | ~256 GB/s |
| Normal DDR5 desktop CPU | ~80 GB/s |
| Normal DDR4 desktop CPU | ~40 GB/s |

That is a roughly 80x range from a normal CPU to an H100, wider than
almost any other hardware variable.

For a 70B model in 4-bit quantization (35 GB of weights):

- **H100**: 96 tokens/sec ceiling
- **M2 Ultra (192GB)**: 23 tokens/sec
- **M4 Max (128GB)**: 15.6 tokens/sec, fits, usable
- **RTX 4090 (24GB)**: doesn't fit on one card
- **DDR5 CPU**: 2.3 tokens/sec, too slow

You see ~70-80% of the ceiling in practice (overhead, KV cache,
attention).

## Why this is specifically the LOCAL constraint, not the cloud constraint

In a datacenter, you batch many users together. With a batch of 32
concurrent users, the inference engine reads each weight once and
uses it for 32 multiplications instead of one. The bandwidth cost is
amortized 32x. Throughput per dollar shoots up. The bottleneck moves
from bandwidth to compute, and the H100's compute throughput (1979
TFLOPS at FP16) starts to matter.

That migration is why H100s are economically interesting in the
cloud. Without batching (which is local AI by definition), the H100's
bandwidth advantage over an RTX 4090 is only 3x for ~50x the price.
The H100 wins for a fleet of users, not for one user.

So the silicon story splits at the batch-size question:

- **Cloud inference (batched)**: compute-bound, FLOPS dominate, H100 / B200 / TPU win
- **Local inference (batch size 1)**: bandwidth-bound, GB/s dominate, Apple Silicon wins because of unified memory, not because of FLOPS

## Why Apple wins locally

Two reasons:

**High bandwidth at consumer prices.** Apple's M-series uses LPDDR5X
soldered onto the SoC package with a very wide bus (512-bit on
Ultras, 256-bit on Maxes). They get HBM-class bandwidth without the
HBM packaging cost because they integrate everything tightly.

**Unified memory means high capacity at any tier.** On an NVIDIA card,
your model has to fit in VRAM, which caps at 24GB on a 4090 and 80GB
on an H100. On an Apple SoC, the GPU sees the same memory as the CPU,
so a 192GB Mac Studio has 192GB of "VRAM" for inference purposes.
Capacity stops being the constraint.

Together: enough capacity to hold a 70B model unquantized, enough
bandwidth to run it at usable speed, in a box the size of a Mac mini.
Nobody else ships that combination at consumer prices.

## Why bandwidth is hard to scale (the permanent constraint)

This explains why bandwidth does not follow Moore's Law the way
compute does. Bandwidth is a packaging problem, not a transistor-
density problem.

To get more bandwidth you need more pins per memory chip, more memory
chips per board, or wider buses. Each costs:

- **Pin count**: physical pins are limited by package size; more pins
  means bigger package and more cost.
- **Wire density**: traces on PCB or interposer have minimum spacing;
  more traces means bigger interposer and exponentially more cost.
- **HBM**: bonded directly to the GPU die via through-silicon vias;
  requires a 2.5D silicon interposer that costs hundreds of dollars
  per chip just for the substrate.
- **Power**: every bit moved across the bus burns energy; high-
  bandwidth memory is also power-hungry.

So bandwidth is roughly priced by the square of how much you want.
This is why a 4090 has 1 TB/s for $1500 and an H100 has 3 TB/s for
$30K. The H100 is paying packaging cost, not silicon cost.

Apple's trick is to skip the discrete-board model entirely and put
everything on one package. That works for laptop / desktop scale. It
does not work for datacenter scale, which is why Apple has zero
presence in the cloud-AI hardware market.

## What this means for sovereignty

The silicon-sovereignty question for local AI has a specific shape:
**what would it take for a non-Apple vendor to ship consumer hardware
with M-class memory bandwidth and capacity?**

AMD's (Project: amd-strix-halo) Ryzen AI Max+ is the first credible
attempt: 96GB unified memory at ~256 GB/s. That is a real opening.
(Project: tenstorrent) is the open-silicon long shot. Everyone else
is either feeding the datacenter or selling small-VRAM consumer GPUs.

Until one of those credibly ships at parity, "local AI" in 2026 is
structurally an Apple ecosystem story plus a small-model
llama.cpp-on-anything story. That is a real existence proof. It is
not a sovereign answer.
